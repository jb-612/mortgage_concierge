ARCHITECT REVIEW  
=================  
Below are the main discrepancies I found between the current implementation and the ADK documentation + best‑practice guides, followed by concrete, step‑oriented change requests for the editor engineer.

───────────────────────────────────────────────────────────────────────────────  
1  Import‑time side effects  
───────────────────────────────────────────────────────────────────────────────  
Finding  
• `mortgage_concierge/agent.py` executes `ingest_bank_docs_to_memory()` at import time. This blocks cold‑start, hides errors and makes tests slower.

Docs / Best‑practice  
• “Do expensive I/O in an explicit bootstrap step, not at import‑time.” (see Runtime → Important behaviours section “Async is Primary”).  

Next steps  
a) Create `mortgage_concierge/bootstrap.py` with an async `init()` that calls `ingest_bank_docs_to_memory()` once.  
b) Remove (or guard) the call from `agent.py`; have CLI / web‑server call `bootstrap.init()` before first use.

───────────────────────────────────────────────────────────────────────────────  
2  Tool design & return schema  
───────────────────────────────────────────────────────────────────────────────  
Findings  
• `search_bank_docs` and `list_loan_tracks` omit the recommended `status` key; error flow and success flow are indistinguishable.  
• `loan_calculator` family returns mixed top‑level keys (`status`, `data` OR `error_message`).  
• `store_state_tool` writes arbitrary keys → prone to key clashes.

Docs  
• Tools page → “Always return a dict. Include a status key” (§Function Tools → Return Type).  
• State page → “Use prefixes (user:, app:, temp:) to avoid collisions”.

Next steps  
a) For each tool add a thin wrapper ensuring:  
   `{"status": "success", "result": <payload>}`  
   or `{"status": "error", "error_message": <msg>}`.  
   (See examples in docs/tools/function‑tools.md).  
b) In `store_state_tool` restrict keys to one namespace, e.g.:

```
PROFILE_KEY = "user:borrower_profile"
tool_context.state[PROFILE_KEY] = {** tool_context.state.get(PROFILE_KEY, {}),
                                   ** incoming_state }
```

c) Update prompt in `prompt.py` to reference the new canonical state key (matches §State → Best Practices).

───────────────────────────────────────────────────────────────────────────────  
3  State model enforcement  
───────────────────────────────────────────────────────────────────────────────  
Finding  
• `BorrowerProfile` pydantic model exists, but nothing validates the data placed in state.

Docs  
• Agents page → `output_key` saves text; for structured data prefer pydantic validation.  
• Callbacks page → After‑tool callbacks are ideal for validation.

Next steps  
a) Add an `after_tool_callback` on `store_state_tool` that:  
   – loads what was just stored;  
   – attempts `BorrowerProfile.model_validate()`;  
   – returns an error `Content` if validation fails.  
b) Register this callback in `root_agent`.

───────────────────────────────────────────────────────────────────────────────  
4  Session/state prefixing  
───────────────────────────────────────────────────────────────────────────────  
Finding  
• All keys are currently “flat” (e.g. `last_advice`).  
• Docs discourage this for persistent session services (§State → Prefixes).

Next steps  
a) Refactor keys:  
   `last_advice`                 → `user:last_advice`  
   `last_city_checked_stateful`  → `temp:last_city_checked_stateful` (if really transient)  
   `guardrail_*`                 → `temp:` prefixed.

Update references accordingly in tools/callbacks.

───────────────────────────────────────────────────────────────────────────────  
5  Instruction completeness  
───────────────────────────────────────────────────────────────────────────────  
Finding  
• `root_agent.instruction` lists some tools but not the OpenAPI‑generated ones.  
• Docs (“Referencing Tool in Agent’s Instructions”) emphasise that the LLM should know when to call each tool.

Next steps  
a) Programmatically append a bulleted list of tool names + one‑liner description to the instruction after `load_loan_calculator_api_tools()` returns.  
   E.g.:

```python
openapi_tools = load_loan_calculator_api_tools()
instruction_suffix = "\n\nAvailable calculator endpoints:\n" + "\n".join(
    f"• {t.name} – {t.description}" for t in openapi_tools)
AGENT_INSTRUCTION += instruction_suffix
```

───────────────────────────────────────────────────────────────────────────────  
6  Global singletons & concurrency  
───────────────────────────────────────────────────────────────────────────────  
Finding  
• `memory_service` and `session_service` are module‑level globals.  
• This precludes using multiple independant agents inside the same process.

Docs  
• Runtime → note about global services being acceptable for *examples* but advise scoped service injection.

Next steps  
a) Pass services explicitly into `bootstrap.init()` and into `root_agent` via dependency injection (or factory).  
b) Remove imports of `memory_store.memory_service` inside tools; accept a `ToolContext` parameter and call `tool_context.memory_service` (already exposed).

───────────────────────────────────────────────────────────────────────────────  
7  OpenAPI tool generation edge cases  
───────────────────────────────────────────────────────────────────────────────  
Finding  
• `openapi_tools.load_loan_calculator_api_tools()` rewrites `servers[0]` unconditionally; may break multi‑server specs.  
• `spec_dict` is passed but `spec_str_type` omitted → future ADK versions may enforce one‑of rule.

Next steps  
a) Add safe guard:

```python
if base_url:
    for srv in spec_dict.get("servers", []):
        srv["url"] = base_url
```

b) Upgrade to latest ADK and pass only `spec_dict` (no `spec_str`), since that path is documented (docs/tools/openapi‑tools.md).

───────────────────────────────────────────────────────────────────────────────  
8  Testing gaps  
───────────────────────────────────────────────────────────────────────────────  
Finding  
• No tests cover the OpenAPI tools or memory search tool.

Docs  
• Evaluate → encourage unit tests via `.test.json`.

Next steps  
a) Add `tests/integration/openapi_tools.test.json` including a mock of `/api/calculate-loan`.  
b) Add a memory search test verifying at least one doc returns expected snippet.
